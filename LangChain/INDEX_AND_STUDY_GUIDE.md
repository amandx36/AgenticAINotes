## 1. Models

LangChain gives you **one unified interface** to talk to any LLM — OpenAI, Anthropic, Google, etc. You don't need to learn each company's API separately.

### Two types of models

| | LLM (Base Model) | Chat Model |
|---|---|---|
| **Input** | Plain text string | List of messages |
| **Output** | Plain text string | AI message object |
| **Role awareness** | No | Yes (system / user / assistant) |
| **Examples** | GPT-3, LLaMA 2, Mistral | GPT-4, Claude, Gemini |
| **Use case** | Text gen, translation | Chatbots, assistants |

### Embedding Models
Convert text → list of numbers (a **vector**). These vectors capture the *meaning* of text so you can compare similarity between two pieces of text. Used in semantic search and RAG.

```
"Hello world"  →  Embedding Model  →  [0.23, -0.11, 0.87, ...]
```

## 2. Prompts

A prompt is the instruction you give to the model. LangChain uses **Prompt Templates** so you can reuse the same structure with different values filled in at runtime.

### Why not just use f-strings?
- **Validation** — LangChain checks you filled all required variables before sending to the model.
- **Ecosystem** — Templates plug into chains, agents, and other LangChain components cleanly.

### PromptTemplate (for LLMs)
```python
from langchain.prompts import PromptTemplate

template = PromptTemplate(
    input_variables=["topic"],
    template="Explain {topic} in simple terms."
)
prompt = template.format(topic="neural networks")
```

### ChatPromptTemplate (for Chat Models)
Used when you need a list of messages with roles.
```python
from langchain.prompts import ChatPromptTemplate

template = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    ("human", "Tell me about {topic}")
])
```

### MessagesPlaceholder
A special slot inside a ChatPromptTemplate where you inject a list of past messages at runtime. This is how you pass **conversation history** to the model.

```python
from langchain.prompts import ChatPromptTemplate, MessagesPlaceholder

template = ChatPromptTemplate.from_messages([
    ("system", "You are a helpful assistant."),
    MessagesPlaceholder(variable_name="chat_history"),
    ("human", "{input}")
])
```

### Three types of messages
| Type | Role | Purpose |
|---|---|---|
| `SystemMessage` | system | Sets the model's behavior/persona |
| `HumanMessage` | user | What the user says |
| `AIMessage` | assistant | What the model replied |

## 3. Runnables

A **Runnable** is LangChain's standard building block. Any component — prompt, model, parser, custom function — that takes an input, processes it, and returns an output is a Runnable.

> **Think of it as a Lego brick.** All bricks have the same connectors, so they snap together in any combination.

### Every Runnable has these methods
| Method | What it does |
|---|---|
| `.invoke()` | Run once, get one result |
| `.stream()` | Get result token by token (streaming) |
| `.batch()` | Run on a list of inputs at once |
| `.ainvoke()` | Async version of invoke |

### Types of Runnables

**RunnableLambda** — wraps any Python function
```python
from langchain_core.runnables import RunnableLambda
double = RunnableLambda(lambda x: x * 2)
```

**RunnableSequence** — steps run one after another (pipe operator `|`)
```python
chain = prompt | llm | output_parser
```

**RunnableParallel** — run multiple Runnables on the same input at the same time
```python
from langchain_core.runnables import RunnableParallel
parallel = RunnableParallel({
    "summary": summary_chain,
    "keywords": keyword_chain
})
```

**RunnablePassthrough** — passes input unchanged, useful when you need the original input alongside transformed output
```python
from langchain_core.runnables import RunnablePassthrough
chain = RunnablePassthrough() | llm
```

**RunnableBranch** — conditional logic, pick which path to run based on the input
```python
from langchain_core.runnables import RunnableBranch
branch = RunnableBranch(
    (lambda x: x["lang"] == "fr", french_chain),
    default_chain  # fallback
)
```

## 4. Chains

A **chain** connects multiple Runnables together so the output of one becomes the input of the next.

```
Prompt Template  →  LLM  →  Output Parser
```

```python
chain = prompt | llm | output_parser
result = chain.invoke({"topic": "gravity"})
```

### Chain vs Agent — key difference

| | Chain | Agent |
|---|---|---|
| **Flow** | Fixed sequence (you decide upfront) | Dynamic (LLM decides at runtime) |
| **Predictability** | High | Lower |
| **Tool usage** | No | Yes |
| **Use when** | Steps are known in advance | Steps depend on the situation |

## 5. Memory

LLM API calls are **stateless** — the model forgets the previous conversation every time. Memory solves this by storing and injecting conversation history.

### Types of Memory

| Memory Type | How it works | Best for |
|---|---|---|
| `ConversationBufferMemory` | Stores every message | Short conversations |
| `ConversationBufferWindowMemory` | Keeps only the last N messages | Medium conversations |
| `ConversationSummaryMemory` | Summarizes old messages using LLM | Long conversations |
| Custom Memory | You define what to store and how | Special use cases |

```python
from langchain.memory import ConversationBufferMemory

memory = ConversationBufferMemory(return_messages=True)
memory.save_context({"input": "Hi"}, {"output": "Hello! How can I help?"})
```

## 6. RAG

**RAG = Retrieval-Augmented Generation**

Instead of relying only on what the LLM learned during training, RAG lets you give it fresh, specific information at query time (like your own documents or database).

### The 4-step RAG pipeline

```
1. Indexing  →  2. Retrieval  →  3. Augmentation  →  4. Generation
```

### Step 1: Indexing (done once, upfront)
Prepare your knowledge base so it can be searched later.

**a) Document Loading** — load data from different sources into a standard `Document` object:
```python
# Standard format every loader produces:
Document(
    page_content="The actual text...",
    metadata={"source": "file.pdf", "page": 1}
)
```

| Loader | Use case |
|---|---|
| `TextLoader` | Plain .txt files |
| `PyPDFLoader` | PDF files (text-based, not scanned) |
| `WebBaseLoader` | Web pages |
| `CSVLoader` | CSV files |
| `DirectoryLoader` | Load all files in a folder |

**b) Text Splitting** — break large documents into smaller chunks the LLM can handle

| Strategy | How it works |
|---|---|
| Length-based | Split every N characters |
| Recursive (most common) | Try splitting by `\n\n`, then `\n`, then words, then characters |
| Structure-based | Use language-specific separators (e.g., Python functions) |
| Semantic | Split based on meaning change (embedding-based) |

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(docs)
```

**c) Embedding & Storage** — convert chunks to vectors and store in a vector database

```python
from langchain.vectorstores import Chroma
from langchain.embeddings import OpenAIEmbeddings

vectorstore = Chroma.from_documents(chunks, OpenAIEmbeddings())
```

> Why a vector DB? Normal SQL databases can't do similarity search on vectors. Vector DBs like Chroma, Pinecone, Weaviate, and Milvus are built exactly for this.

### Step 2: Retrieval
At query time, convert the user's question to a vector and find the most similar chunks.

```
User query  →  Embed query  →  Compare with stored vectors  →  Return top-k matches
```

### Types of Retrievers

| Retriever | What it does |
|---|---|
| `VectorStoreRetriever` | Basic similarity search (most common) |
| `MMR` (Max Marginal Relevance) | Returns diverse results, avoids repetition |
| `MultiQueryRetriever` | LLM generates multiple versions of the query, merges results |
| `WikipediaRetriever` | Fetches from Wikipedia API |

> All retrievers in LangChain are Runnables.

### Step 3: Augmentation
Combine the user's query + retrieved chunks into one prompt.

```python
prompt = f"""Use the context below to answer the question.

Context: {retrieved_chunks}

Question: {user_query}
"""
```

### Step 4: Generation
Send the augmented prompt to the LLM and get the answer.

### Complete RAG chain example
```python
from langchain_core.runnables import RunnablePassthrough

rag_chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt
    | llm
    | output_parser
)

answer = rag_chain.invoke("What is the refund policy?")
```

## 7. Tools & Agents

### What is a Tool?
A Tool is a Python function **packaged in a way the LLM can understand and call when needed**. The LLM doesn't get the actual function — it gets a JSON schema describing what the function does and what inputs it expects.

```python
print(my_tool.args_schema.model_json_schema())
# Shows the LLM what the tool looks like
```

### Types of Tools
- **Built-in tools** — DuckDuckGo search, Shell tool, Wikipedia, etc.
- **Custom tools** — your own functions (call your API, database, business logic)

### Method 1: @tool decorator (simplest)
```python
from langchain.tools import tool

@tool
def get_weather(city: str) -> str:
    """Get the current weather for a given city."""
    # your logic here
    return f"Weather in {city}: Sunny, 25°C"
```
Requirements:
- Add the `@tool` decorator
- Add a docstring (the LLM reads this to understand what the tool does)
- Add type hints for input and return

### Method 2: StructuredTool + Pydantic (for complex inputs)
```python
from langchain.tools import StructuredTool
from pydantic import BaseModel

class SearchInput(BaseModel):
    query: str
    max_results: int = 5

def search(query: str, max_results: int) -> str:
    """Search the web."""
    ...

search_tool = StructuredTool.from_function(
    func=search,
    args_schema=SearchInput
)
```

### Method 3: BaseTool class (for async tools)
```python
from langchain.tools import BaseTool

class MyTool(BaseTool):
    name = "my_tool"
    description = "Does something useful"

    def _run(self, query: str) -> str:
        return "result"

    async def _arun(self, query: str) -> str:
        return "async result"
```

### Toolkits
A **toolkit** is a collection of related tools that serve a common purpose (e.g., a SQL toolkit has tools for running queries, describing tables, etc.).

---

### Tool Binding, Tool Calling, Tool Execution

These are 3 different steps — don't mix them up.

**1. Tool Binding** — register tools with the LLM so it knows what's available
```python
llm = ChatOpenAI()
llm_with_tools = llm.bind_tools([get_weather, search_tool])
```

**2. Tool Calling** — the LLM *suggests* which tool to call and with what arguments. **The LLM does NOT actually run the tool**, it just returns a suggestion.

**3. Tool Execution** — your code actually runs the function with the LLM's suggested arguments.

**4. Tool Message** — the result of execution is sent back to the LLM as a `ToolMessage` so it can continue.

---

### What is an Agent?
An AI agent is an intelligent system that:
- Receives a high-level goal from the user
- Breaks it into steps autonomously
- Decides which tools to call (and when)
- Maintains context across multiple steps
- Adapts based on new information

> **Agent** = plans + thinks (the LLM's reasoning part)
> **Agent Executor** = actually runs the tools and actions

### Agent vs Chain
| | Agent | Chain |
|---|---|---|
| **Decision making** | Dynamic — LLM decides | Fixed — you decide |
| **Tools** | Can use tools | Cannot |
| **Steps** | Unknown upfront | Known upfront |

### ReAct Pattern (most common agent pattern)
**Re**asoning + **Act**ing — the agent alternates between thinking and doing:
```
Thought: I need to find the weather in Paris
Action: get_weather("Paris")
Observation: Sunny, 22°C
Thought: I have the answer
Final Answer: It's sunny and 22°C in Paris
```

## 8. Output Parsers

**Structured Output** means getting the LLM's response in a well-defined format (like JSON or a Python object) instead of raw text.

Used for: data extraction, building APIs, feeding output into other systems.

### Types of Output Parsers

| Parser | What it does | Use when |
|---|---|---|
| `StrOutputParser` | Returns plain text string | You just need the text |
| `JsonOutputParser` | Parses JSON from LLM response | You need a dict/list |
| `PydanticOutputParser` | Validates and parses into a Pydantic model | You need strict types + validation |
| `StructuredOutputParser` | Returns dict based on response schema | Simple structured output (no validation) |

### StrOutputParser (simplest)
```python
from langchain_core.output_parsers import StrOutputParser

chain = prompt | llm | StrOutputParser()
result = chain.invoke({"topic": "AI"})  # returns a plain string
```

### PydanticOutputParser (most robust)
```python
from langchain.output_parsers import PydanticOutputParser
from pydantic import BaseModel, Field

class Person(BaseModel):
    name: str = Field(description="Person's full name")
    age: int = Field(description="Person's age")
    email: str = Field(description="Person's email address")

parser = PydanticOutputParser(pydantic_object=Person)

prompt = PromptTemplate(
    template="Extract person info.\n{format_instructions}\n{text}",
    input_variables=["text"],
    partial_variables={"format_instructions": parser.get_format_instructions()}
)

chain = prompt | llm | parser
person = chain.invoke({"text": "John is 30 years old. Email: john@example.com"})
# Returns a Person object with name="John", age=30, email="john@example.com"
```

### with_structured_output (easiest modern approach)
```python
from pydantic import BaseModel

class Invoice(BaseModel):
    vendor: str
    amount: float
    date: str

structured_llm = llm.with_structured_output(Invoice)
result = structured_llm.invoke("Extract invoice: Amazon charged $49.99 on Jan 5")
# Returns an Invoice object directly
```

Two modes under the hood:
- **Function calling** — model returns a structured JSON natively (preferred)
- **JSON mode** — model is told to always return valid JSON

## 9. Interview Q&A

**Q: What is LangChain?**
> LangChain is a framework for building applications with LLMs. It provides standard components — prompts, models, chains, memory, tools, agents — that all work together through a unified Runnable interface.

---

**Q: What is a Runnable?**
> A Runnable is LangChain's standard interface for any unit of work. It takes an input, processes it, and returns an output. Every Runnable supports the same methods — invoke, stream, batch — so you can chain them together using the pipe operator `|`, run them in parallel, or swap implementations without changing your code.

---

**Q: What's the difference between a Chain and an Agent?**
> A chain follows a fixed sequence of steps that you define at write time. An agent uses an LLM to dynamically decide what steps to take at runtime — including which tools to call and in what order.

---

**Q: What is RAG and why is it useful?**
> RAG stands for Retrieval-Augmented Generation. Instead of relying only on what the LLM knows from training, RAG retrieves relevant information from your own documents at query time and passes it to the model as context. This is useful for answering questions about private or recent data the model wasn't trained on.

---

**Q: What is the difference between Tool Binding, Tool Calling, and Tool Execution?**
> - **Tool Binding** = registering tools with the LLM so it knows what's available
> - **Tool Calling** = the LLM suggesting which tool to use and with what arguments (the LLM does NOT run the tool itself)
> - **Tool Execution** = your code actually running the function with the LLM's suggested arguments

---

**Q: Why can't we store embeddings in a regular SQL database?**
> Regular SQL databases don't support similarity search on high-dimensional vectors. Vector databases like Chroma, Pinecone, and Weaviate are specifically built for this — they can efficiently find the most similar vectors using algorithms like HNSW or FAISS.

---

**Q: What is the difference between LLMs and Chat Models?**
> LLMs take plain text in and return plain text out. Chat Models take a list of messages (with roles: system, user, assistant) and return a message. Chat models are instruction-tuned and understand conversational context. Most modern models you'd use (GPT-4, Claude, Gemini) are chat models.

---

**Q: Why use Pydantic for output parsing?**
> Pydantic gives you strict schema enforcement, type safety, and automatic validation. If the LLM returns the wrong type (e.g., a string instead of an integer), Pydantic raises an error rather than silently passing bad data through your pipeline.

---

**Q: What is MessagesPlaceholder?**
> It's a special slot inside a ChatPromptTemplate where you inject a list of messages at runtime. You use it to pass conversation history to the model without hardcoding it into the template.

---

**Q: What is the ReAct agent pattern?**
> ReAct stands for Reasoning + Acting. The agent alternates between thinking about what to do (Thought), performing an action (Action), and observing the result (Observation) — repeating until it reaches a final answer. This pattern makes the agent's reasoning transparent and debuggable.

---

**Q: What is chunking and why does it matter?**
> Chunking is breaking large documents into smaller pieces before embedding them. It matters because: (1) LLMs have context length limits, (2) smaller, focused chunks lead to more precise retrieval, (3) embedding a 50-page PDF as one chunk means you always retrieve the whole thing — not the relevant part.

---

**Q: What is MMR retrieval?**
> MMR stands for Maximum Marginal Relevance. Instead of returning the top-k most similar chunks (which might all say the same thing), MMR balances relevance with diversity — so you get varied, non-redundant results.

## 10. Quick One-Liners to Memorize

| Concept | One-liner |
|---|---|
| **Runnable** | Any component that takes input, processes, returns output — and chains with `\|` |
| **Chain** | Fixed pipeline: A → B → C using the pipe operator |
| **Agent** | LLM that decides its own steps and picks tools dynamically |
| **RAG** | Give the LLM fresh context from your docs at query time |
| **Tool** | Python function packaged as JSON schema so LLM can suggest calling it |
| **Embedding** | Text converted to a vector that represents its meaning |
| **Vector DB** | Database built for similarity search on embeddings |
| **PromptTemplate** | Reusable prompt with variable slots |
| **Memory** | Stores conversation history and injects it into future prompts |
| **Output Parser** | Converts raw LLM text output into structured Python objects |
| **Tool Binding** | Registering tools with the LLM |
| **Tool Calling** | LLM suggesting which tool to use (not running it) |
| **Chunking** | Splitting large docs into small pieces before embedding |

## Bonus: Architecture Flows

### Basic Chain
```
User Input  →  PromptTemplate  →  LLM  →  OutputParser  →  Result
```

### RAG Pipeline
```
[Indexing]
Docs  →  Loader  →  TextSplitter  →  EmbeddingModel  →  VectorStore

[Query time]
User Query  →  Embed  →  VectorStore.search()  →  Top-k Chunks
           ↓
Prompt (query + chunks)  →  LLM  →  Answer
```

### Agent Loop
```
User Goal
    ↓
LLM thinks: "What should I do?"
    ↓
Tool Call suggestion
    ↓
Tool Execution (actual function runs)
    ↓
Result sent back to LLM as ToolMessage
    ↓
LLM thinks again → more tool calls OR Final Answer
```
